## 三、刷新页面时缩略图闪烁两次

### 问题描述

原实现使用 `new Image()` 对象探测文件存在性，每次探测都会发出完整图片加载请求。探测过程中浏览器先渲染 HTML 中的默认 img src（第一次显示），异步探测完成后 `initHomepageThumbnails` 将 img.src 替换为随机选中的图片（第二次加载），导致用户看到图片闪烁/替换两次。

### 修复内容

修改文件：`src/renderer/js/thumbnail-selector.js`

将 `checkFileExists()` 从 Image 对象探测改为 `fetch HEAD` 请求：

```javascript
async function checkFileExists(url) {
  try {
    const resp = await fetch(url, { method: 'HEAD' });
    return resp.ok;
  } catch {
    return false;
  }
}
```

HEAD 请求只获取 HTTP 头信息，不加载图片体，速度极快且不产生视觉副作用。

## 四、部分样式导出遗失焦距单位 mm

### 问题描述

`src/renderer/js/exif.js` 中的 `getFocalLength()` 函数返回原始 EXIF 值。`FocalLengthIn35mmFilm` 可能返回纯数字（如 `50`，无 "mm" 后缀），而 `FocalLength` 经过 `exifKeyFormatter` 处理后可能带 "mm"（如 `50mm`）。

部分导出文件（如 Type A、B、C、D export）直接使用 `settings.focalLength` 而不做 `.replace(/mm$/i, '')` + 重新添加 "mm" 的处理，导致当值为纯数字时导出图片上缺少 "mm" 单位。

### 修复内容

修改文件：`src/renderer/js/exif.js`

在 `getFocalLength()` 中统一确保返回值带 "mm" 后缀：

```javascript
export function getFocalLength(exif) {
  let focal;
  if (exif['FocalLengthIn35mmFilm']) {
    focal = exif.FocalLengthIn35mmFilm;
  } else if (exif.FocalLength) {
    focal = exif.FocalLength;
  }
  if (!focal) return null;
  const str = String(focal).trim();
  return str.endsWith('mm') ? str : `${str}mm`;
}
```

已做 `.replace(/mm$/i, '')` + 重新添加 "mm" 的样式（Type E、F、G、H、I、J、K、L、M）不受影响，因为它们会先去掉再加回。

## 六、V1.13 后续优化（首页缩略图无闪烁改进）

### 问题描述

V1.13 实现中，HTML img src 直接指向 Sample/ 目录下的实际图片文件。虽然 `initHomepageThumbnails` 会异步替换为随机选中的图片，但在随机化完成前，浏览器会先加载并显示 HTML 中的默认图片。随机化完成后 img.src 被替换，浏览器再次加载新图片，导致用户看到短暂的默认图闪现。

### 修复内容

修改文件：`src/renderer/index.html`

- 将 13 个样式卡片的 img src 从 `Sample/{NNN}-TypeX-sample_compressed.jpeg` 改为 **透明 GIF data URI 占位符**：
  ```html
  src="data:image/gif;base64,R0lGODlhAQABAIAAAAAAAP///yH5BAEAAAAALAAAAAABAAEAAAIBRAA7"
  ```
- 将原始 Sample 路径移至新增的 `data-fallback-src` 属性：
  ```html
  data-fallback-src="Sample/022-TypeA-sample_compressed.jpeg"
  ```
- 这样页面初始加载时样式卡片显示透明区域，随机化完成后一次性写入最终图片，完全消除闪烁。

修改文件：`src/renderer/js/thumbnail-selector.js`

- `buildStyleThumbnailMeta()` 函数增加 `data-fallback-src` 优先读取逻辑：
  ```javascript
  const fallbackSrc = img.getAttribute('data-fallback-src') || '';
  const src = fallbackSrc || img.getAttribute('src') || '';
  ```
- 确保回退路径仍能正确提取样式类型名称（TypeA 等）。

### 效果

- 页面加载时样式卡片区域为透明（无闪烁）
- `initHomepageThumbnails` 完成后一次性写入随机缩略图
- 若随机化失败或超时，卡片保持透明（不显示错误图片）

## 七、性能优化（manifest 清单法）

### 原问题

V1.13 初始实现的 `thumbnail-selector.js` 使用 **暴力探测法** 确定哪些样本图片实际存在：

1. `getDefaultImageIdList(99)` 生成 `'001'` 到 `'099'` 共 99 个候选 ID
2. 13 个样式通过 `Promise.all` 并行执行
3. 每个样式**内部串行**遍历 99 个 ID，每个 ID 先尝试 `.jpeg` 再尝试 `.jpg`
4. 每次尝试发起一个 `fetch(url, { method: 'HEAD' })` 请求

**最坏情况请求数：13 样式 × 99 ID × 2 扩展名 = 2,574 次 HTTP HEAD 请求**

| 场景 | 影响 |
|------|------|
| Cloudflare Pages 部署 | 每个 HEAD 请求经过 Cloudflare 边缘，免费版有请求限制（100K/天），首次探测需回源，延迟高 |
| 局域网 NAS 部署 | 2,574 次请求导致首页加载缓慢，NAS 设备 I/O 压力大 |
| 移动端 | 高延迟网络下体验极差，可能超时 |

### 优化方案：manifest 清单法

用 **1 次 fetch 请求**替代 2,574 次 HEAD 探测。

新建文件：`src/renderer/Sample/sample-manifest.json`

维护一个静态 JSON 清单文件，列出所有实际存在的样本文件按样式分组：

```json
{
  "version": 1,
  "generated": "2026-07-02",
  "samples": {
    "TypeA": ["001", "022", "032"],
    "TypeB": ["002", "017", "028"],
    ...
  }
}
```

当前共 33 个样本文件，覆盖全部 13 种样式。

### 实现细节

重写文件：`src/renderer/js/thumbnail-selector.js`（264 行 → 163 行）

**新增函数：**

- `fetchManifest(manifestUrl)` — 获取 `sample-manifest.json` 清单，返回解析后的 JSON 对象或 null
- `selectRandomFromManifest(manifest, metaList)` — 从清单中为每个样式随机选取缩略图，Fisher-Yates 洗牌 + 全局 imageId 去重

**移除函数：**

- `getDefaultImageIdList()` — 不再需要遍历 99 个 ID
- `collectCandidatesForStyle()` — 不再需要逐个探测
- `assignUniqueIdThumbnails()` — 逻辑合并到 `selectRandomFromManifest()`
- `checkFileExists()` — 不再作为主要探测手段

**主入口 `initHomepageThumbnails()` 流程变更：**

```
旧流程：遍历 99 ID × 2 扩展名 × 13 样式 → 2,574 次 HEAD 请求
新流程：fetch manifest.json → 内存随机选取 → 写入 img.src（1 次请求）
```

**回退机制：** manifest 获取失败时，使用 `data-fallback-src` 作为回退图片。

**API 兼容：** `initHomepageThumbnails(styleCards)` 签名和返回值不变，`app.js` 无需修改。

### 性能对比

| 指标 | 优化前（暴力探测） | 优化后（manifest） |
|------|-------------------|-------------------|
| HTTP 请求数 | 最多 2,574 次 HEAD | **1 次 GET** |
| 首页加载时间（Cloudflare） | 10-30 秒 | **< 1 秒** |
| 首页加载时间（局域网 NAS） | 3-8 秒 | **< 0.5 秒** |
| manifest 文件大小 | — | ~500 bytes |

## 八、manifest ID 格式修正

### 问题描述

`sample-manifest.json` 初版中 TypeA 的 ID 写成了 `"01"`（2 位），但实际文件名使用 3 位零填充格式 `001-TypeA-sample_compressed.jpeg`。浏览器请求 `Sample/01-TypeA-sample_compressed.jpeg` 返回 404。

### 修正内容

将 manifest 中 TypeA 的 ID 从 `"01"` 修正为 `"001"`：

```json
// 修正前
"TypeA": ["01", "022", "032"]

// 修正后
"TypeA": ["001", "022", "032"]
```

### 维护说明

当添加或删除 Sample 目录中的样本图片后，需要更新 `sample-manifest.json`。可用以下命令自动生成：

```powershell
$files = Get-ChildItem src/renderer/Sample/*_compressed.jpeg -Name
$samples = @{}
foreach ($f in $files) {
  if ($f -match '^(\d+)-Type([A-Z])-sample_compressed\.jpeg$') {
    $id = $matches[1]; $type = "Type$($matches[2])"
    if (-not $samples[$type]) { $samples[$type] = @() }
    $samples[$type] += $id
  }
}
@{ version = 1; generated = (Get-Date -Format 'yyyy-MM-dd'); samples = $samples } |
  ConvertTo-Json -Depth 3 | Set-Content src/renderer/Sample/sample-manifest.json -Encoding UTF8
```
