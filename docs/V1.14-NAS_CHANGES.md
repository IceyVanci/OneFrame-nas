## 一、字体 woff2 优化

### 问题描述

MiSans 字体文件为原始 TTF 格式，3 个文件共 **23.4MB**（Normal 7.9MB、Medium 7.9MB、Semibold 7.8MB）。CSS `@font-face` 未声明 `font-display`，浏览器默认使用 `block` 行为——遇到使用 MiSans 的文字时**阻塞渲染最多 3 秒**，等待字体下载完成。

Cloudflare 部署场景下，首次访问需回源获取 23MB 字体，延迟可能 2-5 秒。

### 修复内容

修改文件：`src/renderer/index.css`

**1. 切换 woff2 优先，TTF 作为回退：**

```css
@font-face {
  font-family: 'MiSans';
  src: url('fonts/MiSans-Normal.woff2') format('woff2'),
       url('fonts/MiSans-Normal.ttf') format('truetype');
  font-weight: normal;
  font-style: normal;
  font-display: swap;
}
```

woff2 压缩率约 38%，3 个文件从 23.4MB 降至 **14.5MB**。

**2. 添加 `font-display: swap`：**

浏览器先用系统字体（Microsoft YaHei）渲染页面，字体加载完成后自动替换。页面不被字体阻塞。

**3. nginx 长期缓存（已有，无需修改）：**

```nginx
location ~* \.(ttf|woff2?|eot|otf)$ {
    add_header Access-Control-Allow-Origin "*";
    add_header Cache-Control "public, max-age=31536000, immutable";
}
```

woff2 匹配 `woff2?` 正则，CORS 和长期缓存自动生效。二次访问零延迟。

### 字体文件保留说明

| 文件 | 能否删除 | 原因 |
|------|---------|------|
| `MiSans-*.woff2` | 不能 | 网页预览的字体来源（CSS @font-face） |
| `MiSans-*.ttf` | 不能 | 13 个导出模块通过 `opentype.load()` 加载 TTF 进行 Canvas 文字绘制 |

两套文件各司其职：woff2 加速页面渲染，TTF 支持 Canvas 导出。

## 二、首页加载动画（Shimmer）

### 问题描述

V1.13.1 使用透明 GIF 占位符 + manifest 清单法后，页面加载时样式卡片区域为透明，与深色背景（`#0a0a0a`）融为一体，用户看到黑屏。从 manifest fetch 到图片加载完成有一段空白期。

### 修复内容

**修改文件：`src/renderer/index.css`**

添加 CSS shimmer（微光）骨架屏动画：

```css
.style-card .preview-image.loading {
  background: linear-gradient(110deg, #1a1a1a 8%, #2a2a2a 18%, #1a1a1a 33%);
  background-size: 200% 100%;
  animation: shimmer 1.5s ease-in-out infinite;
}

@keyframes shimmer {
  0% { background-position: 200% 0; }
  100% { background-position: -200% 0; }
}
```

**修改文件：`src/renderer/index.html`**

- 13 个样式卡片 img 添加 `loading` class
- `<head>` 中添加 manifest 预加载：
  ```html
  <link rel="preload" href="Sample/sample-manifest.json" as="fetch" crossorigin>
  ```

**修改文件：`src/renderer/js/thumbnail-selector.js`**

写入 img.src 时添加 `loading` class，图片加载完成后自动移除：

```javascript
img.classList.add('loading');
const cleanup = () => img.classList.remove('loading');
img.addEventListener('load', cleanup, { once: true });
img.addEventListener('error', cleanup, { once: true });
img.src = path;
```

### 效果

- 页面加载时样式卡片显示灰色脉冲动画（非黑屏）
- 缩略图加载完成后 shimmer 自动消失
- manifest 预加载减少等待时间

## 三、样本图片压缩

### 问题描述

Sample 目录中的样本图片未经优化压缩，167 个文件总大小约 **34MB**，影响 Cloudflare Pages 部署的构建时间和首次加载速度。

### 压缩参数

- **分辨率**：从原始 1714×1280 缩放至 **1200px 宽**（保持原始比例）
- **JPEG 质量**：q:v 8（约 80% 质量）
- **压缩命令**：`ffmpeg -vf "scale=1200:-1" -q:v 8`

### 压缩结果

| 指标 | 压缩前 | 压缩后 |
|------|--------|--------|
| 总文件数 | 167 | 167 |
| 总大小 | ~34 MB | **16.4 MB** |
| 单文件大小 | 200-420 KB | 60-130 KB |
| 压缩率 | — | **52%** |

原图备份在 `src/renderer/Sample_backup/`。

## 五、Manifest 自动生成机制

### 问题描述

`sample-manifest.json` 需要与 Sample 目录中的实际文件保持同步。之前手动维护容易遗漏（如新增/删除样本图片后忘记更新 manifest），导致 404 错误。

### 解决方案

在 Docker 构建流程中自动扫描 Sample 目录生成 manifest，无需手动维护。

**新建文件：`scripts/generate-manifest.sh`**

Shell 脚本，在 Docker 构建时运行（alpine 环境）：
- 扫描 Sample 目录中所有 `*-TypeX-sample_compressed*` 文件
- 按样式类型分组提取 imageId
- 输出标准 JSON 格式的 `sample-manifest.json`

**修改文件：`docker/Dockerfile`**

在 COPY 样本图片之后、COPY nginx 配置之前添加 manifest 生成步骤：

```dockerfile
COPY scripts/generate-manifest.sh /tmp/generate-manifest.sh
RUN chmod +x /tmp/generate-manifest.sh && /tmp/generate-manifest.sh
```

**修改文件：`.dockerignore`**

排除不需要进入 Docker 镜像的目录：
- `src/renderer/Sample_backup` — 原图备份（~34MB）
- `src/renderer/Samplearchive` — 旧版存档目录

build context 从 60MB 降至 **21KB**。

### 效果

每次 `docker compose up --build` 时，manifest 自动基于 Sample 目录实际文件生成，无需手动维护。

## 七、重选图片 EXIF 显示修复

### 问题描述

点击"重选图片"按钮导入新图片后，EXIF 信息只在编辑面板的输入框中被正确填充，但预览区域的边框内容（Logo、型号、参数等）没有更新。

### 根因分析

`loadImageWithExif()` 函数的执行时序问题：

1. `resetForm()` — 清空所有表单输入
2. `userImage.src = URL.createObjectURL(file)` — 触发 img load 事件
3. img load 事件 → `updateBorder()` → `updateBorderContent()` — **此时表单还是空的！**
4. `await getExif(file)` — 异步解析 EXIF
5. `updateExifDisplay()` — 用 EXIF 数据填充表单输入
6. **但之后没有人再次调用 `updateBorderContent()`** — 预览不会更新

### 修复内容

修改文件：`src/renderer/js/app.js`

在 `loadImageWithExif()` 的 `updateExifDisplay()` 之后添加 `updateBorderContent()` 调用：

```javascript
currentExif = nextExif || {};
updateExifDisplay();
updateBorderContent();  // 新增：EXIF 填充后刷新预览
```
